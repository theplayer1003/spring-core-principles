# 1. 도메인 설계 및 순수 Java DI 컨테이너(AppConfig) 도입

## 1. 목적

- 도메인(Member)및 저장소(Repository) 클래스 설계.
- 클라이언트 코드가 구현체에 의존할 때 발생하는 문제점 확인.
- 관심사의 분리를 위한 DI 컨테이너(AppConfig) 도입.

## 2. 구현 내용

- Entity & Repository
  `Member`, `Grade`, `MemberRepository` (interface), `MemoryMemberRepository` (implementation) 작성
- Service
  `MemberService` (interface), `MemberServiceImpl` (implementation) 작성
- AppConfig
  DI 컨테이너, 객체의 생성과 연결을 담당하는 설정 클래스 작성

## 의존성 주입 리팩토링 과정

```java
public class MemberServiceImpl implements MemberService {

    private final MemberRepository memberRepository = new MemoryMemberRepository();
    /* 
    DIP, 의존관계 역전 원칙 위반
    초기 코드는 고수준 모듈인 MemberServiceImpl 가 MemberRepository 라는 추상화된 인터페이스에 의존하는 것 처럼 보이지만,
    실제로는 new MemoryMemberRepository() 를 통해 구현체 클래스에 강하게 의존하고 있는 상태였다.
    
    OCP, 개방-폐쇄 원칙 위반
    추후 MemoryMemberRepository 가 JdbcMemberRepository 등으로 교체해야한다고 할 경우,
    해당 클래스를 직접 수정해야만 한다.
     */
}
```

```java
public class MemberServiceImpl implements MemberService {

    private final MemberRepository memberRepository;

    public MemberServiceImpl(MemberRepository memberRepository) {
        this.memberRepository = memberRepository;
    }
    /*
    생성자를 통해 외부에서 의존성을 주입받도록 하여 DIP, OCP 문제를 해결한다.
     */
}
```
